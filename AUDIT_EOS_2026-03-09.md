# Audit EOS

Date: 2026-03-09

## Resume

Le projet a une base fonctionnelle reelle: architecture backend/frontend separee, support multi-clients, logique metier riche, exports specialises EOS et PARTNER, et quelques optimisations utiles comme le lazy loading des onglets admin.

En revanche, plusieurs points critiques reduisent fortement la fiabilite du produit:

- securite quasi absente sur l'interface enqueteur
- incoherences frontend/backend sur certaines routes
- endpoints legacy qui cassent a l'execution
- hygiene frontend insuffisante
- configuration de lancement trop liee a la machine locale

## Points positifs

### 1. Separation claire des couches

- `backend/` contient la logique Flask, les routes, services, models, migrations
- `frontend/` contient l'interface React/Vite
- les exports EOS et PARTNER sont separes

### 2. Support multi-clients reel

Le support multi-clients n'est pas seulement cosmetique:

- presence de `client_id` dans plusieurs models
- filtres par client dans plusieurs routes backend
- composants frontend capables de changer de client

### 3. Modele de donnees riche

Le modele `Donnee` contient:

- historique JSON
- liens de contestation vers enquete originale
- statuts de validation
- champs d'export et de facturation
- index SQL utiles pour la recherche et les filtres

### 4. Base de performance correcte cote admin

L'interface admin charge plusieurs onglets en lazy loading, ce qui est une bonne pratique pour reduire le cout initial de chargement.

## Defauts majeurs

### 1. Securite tres faible de l'interface enqueteur

Constat:

- la connexion enqueteur repose seulement sur un email
- aucune authentification forte
- aucune session serveur
- aucune verification d'autorisation cote backend
- l'identite est stockee en `localStorage`

Probleme concret:

- le dashboard enqueteur charge toutes les donnees via `/api/donnees-complete`
- le filtrage par enqueteur se fait ensuite cote frontend

Impact:

- un utilisateur technique peut voir ou appeler bien plus de donnees que prevu
- ce point est critique si l'application sort d'un environnement strictement interne

## 2. Gestion du template VPN cassee

Le flux frontend/backend n'est pas coherent:

- frontend: route `POST /api/vpn/template`
- backend: route `POST /api/vpn-template`

Autres problemes:

- frontend envoie le champ `template`
- backend attend le champ `file`
- la route de telechargement utilise `send_file` sans import

Conclusion:

- la fonctionnalite "Template VPN" est tres probablement inutilisable en l'etat

### 3. Endpoints legacy de validation casses

Deux routes importent un module inexistant:

- `backend/routes/validation.py`
- `backend/routes/enquetes.py`

Elles referencent `models.donnee_enqueteur`, alors que le fichier reel est `models/models_enqueteur.py`.

Impact:

- certains anciens endpoints de validation ou confirmation echoueront a l'execution

### 4. Export Word de l'ecran Assignations trompeur

Le frontend prepare une liste filtree des enquetes selectionnees, mais le backend:

- ignore cette liste
- exporte toutes les enquetes non exportees du client

Impact:

- le comportement affiche ne correspond pas au comportement reel
- risque d'export involontaire de dossiers non vises

### 5. Configuration de lancement trop locale

Le backend force `DATABASE_URL` en dur:

- dans `backend/start_with_postgresql.py`
- dans `DEMARRER_EOS_SIMPLE.bat`

Impact:

- deploiement fragile
- changement d'environnement difficile
- identifiants de base visibles et faibles

### 6. Gestion du schema de base fragile

Le backend appelle encore `db.create_all()` au demarrage alors que le projet utilise Alembic.

Impact:

- risque de derive entre le schema reel et les migrations
- erreurs difficiles a diagnostiquer sur une nouvelle machine ou une base deja existante

### 7. CORS trop permissif

Le backend autorise `origins="*"` en developpement.

Pris seul, ce n'est pas toujours dramatique, mais combine a l'absence d'authentification solide, c'est un mauvais signal de securite.

### 8. Hygiene frontend insuffisante

Verifications realisees:

- `python -m compileall backend`: OK
- `npm run lint`: echec
- `npm run build`: non valide ici a cause de la sandbox (`spawn EPERM`)

Le lint frontend remonte beaucoup de problemes:

- variables inutilisees
- texte JSX non echappe
- hooks React avec dependances manquantes
- code mort

Impact:

- maintenance plus difficile
- risque de regressions plus eleve

### 9. Type MIME incorrect pour le telechargement des batchs EOS

Les exports batch EOS sont servis avec un type MIME de document Word alors qu'il s'agit de fichiers texte.

Impact:

- certains navigateurs ou outils peuvent mal interpreter le fichier

## Fonctions probablement cassees ou non fiables

### Cassees ou tres probablement cassees

- gestion du template VPN
- endpoints legacy de validation
- endpoints legacy de confirmation

### Fonctionnelles mais fragiles ou trompeuses

- interface enqueteur
- export Word depuis l'ecran Assignations
- demarrage sur une autre machine
- gestion des migrations/schema

## Fonctions qui semblent bien pensees

- support multi-clients
- logique PARTNER de mots-cles, demandes et tarifs
- export EOS par batch
- historisation des enquetes
- separation entre validation, export, archives et paiements

## Tests et couverture

Je n'ai pas trouve de vraie suite de tests applicatifs structuree pour le projet.

Constat:

- pas de script `test` dans `frontend/package.json`
- quelques scripts Python nommes `test_...` dans `backend/scripts/`
- rien qui ressemble a une couverture continue propre du produit complet

Impact:

- chaque correction reste risquee
- forte dependance aux tests manuels

## Conclusion

Le projet est loin d'etre vide ou mal pense. Il y a un vrai noyau metier, une logique de production reelle, et une base suffisamment riche pour etre stabilisee.

Mais en l'etat, il ne faut pas le considerer comme robuste.

Priorites les plus importantes:

1. securiser l'interface enqueteur
2. corriger les incoherences de routes frontend/backend
3. supprimer ou reparer les endpoints legacy casses
4. nettoyer la configuration de lancement et la gestion DB
5. remettre un minimum de discipline frontend et de tests

## Verification effectuee

- lecture de la structure du projet
- lecture des routes backend principales
- lecture des composants frontend critiques
- verification des incoherences d'API
- compilation Python locale
- lint frontend local

## Limites de cet audit

- pas de parcours complet sur base PostgreSQL reelle
- pas de test fonctionnel navigateur complet
- build frontend non confirme dans cette sandbox a cause d'un blocage `esbuild`
